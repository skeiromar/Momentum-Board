# Internationalization (i18n)

The boilerplate includes full i18n support via react-intl. English is the fallback language: if a translation is missing for a locale, the English text is used. Same idea as login: harmless if unused, helpful when used.

## Usage options

1. **Ignore it and never run the script** — Ship in a single language. Keep using `FormattedMessage` with en.json or migrate to plain strings. Don't run `npm run check:i18n`.

2. **Transition to a language with English as default, never run the script** — Use i18n for structure, default to English for missing keys. Add locales as needed; English fills in gaps. Don't run `npm run check:i18n`.

3. **Use it for everything and run the script** — Fully translate all locales. Run `npm run check:i18n` before commit to ensure every locale has all keys. See [scripts/README.md](../scripts/README.md).

## Overview

- **Library:** react-intl
- **Locale files:** `src/client/locales/` (en.json, ar.json, fr.json, zh.json)
- **Provider:** `I18nProvider` in `src/client/utilities/i18n.tsx` wraps the app and reads locale from Redux preferences
- **RTL:** Arabic (`ar`) automatically switches layout to right-to-left
- **Validation:** `npm run check:i18n` (optional; only for option 3) ensures all locale files stay in sync


## Using Messages

### FormattedMessage (JSX)

For inline text in components:

```tsx
import { FormattedMessage } from 'react-intl';

<FormattedMessage id="home.title" />
<FormattedMessage id="footer.copyright" values={{ year: new Date().getFullYear() }} />
```

### formatMessage (imperative)

When you need the string value (e.g. for attributes like `title` or `aria-label`):

```tsx
import { useIntl } from 'react-intl';

const intl = useIntl();
title={intl.formatMessage({ id: 'home.feature.fast.title' })}
```

## Adding a New Message

1. Add the key and value to all locale files in `src/client/locales/` (en.json, ar.json, fr.json, zh.json).
2. Use it in your component via `FormattedMessage` or `formatMessage`.
3. If using option 3, run `npm run check:i18n` to validate keys match across locales and all used ids exist.

## Adding a New Locale

1. Create `src/client/locales/<code>.json` with the same keys as en.json.
2. Add the locale to `SUPPORTED_LOCALES` in `src/client/utilities/i18n.tsx`.
3. Add the import and entry in the `MESSAGES` object in the same file.
4. Optionally add the locale to `RTL_LOCALES` if it is right-to-left.
5. Update `scripts/check-i18n-keys.mjs` to include the new file in `localeFiles`.
